# 실용주의 프로그래머

> 실용주의 프로그래머를 읽고, 간단히 기록을 남긴다.

어설픈 변명을 만들지 말고 대안을 제시하라.

변화의 촉매가 되라.

적당히 괜찮은 소프트웨어 : 언제 멈춰야 할지 알라

지식에 대한 투자가 언제나 최고의 이윤을 낸다.

읽고 듣는것을 비판적으로 분석하라.

지역사용자모임에 참여하라

DRY = don’t repeat yourself

    - 개발자간의 적극적이고 빈번한 소통

직교성 : 관련없는것들간에 서로 영향이 없도록 하라

예광탄 : 프로그램의 골격을 미리 작성해보고 방향을 정한다.

코드와 일정은 반복하면서 조정하라

가정하지 마라, 증명하라

완벽한 소프트웨어는 만들 수 없다.

단정문을 사용해 불가능한 상황을 대비하라

    - 꼭 단정문이 아니더라도 방어적인 코드를 심어두면 좋을듯
    - 테스트가 모든 경우를 확인하지는 않기 때문

시작한 것은 끝내라

모듈간의 결합도를 최소화 하라

    - 성능향상과의 줄다리기

결합하지 말고 설정하라.

    - 코드에는 추상화를, 메타데이터에는 세부 내용을,

언제나 동시성을 고려해 설계하여라

모델에서 뷰를 분리하라

    - 데이터와 해석을 분리

우연에 맞기는 프로그래밍을 하지 마라

일찍, 자주 리펙터링 하라.

테스트를 염두해 두고 설계하라

    - 테스트 주도 개발 참고

자신이 이해하지 못하는, 마법사가 만들어 준 코드는 사용하지 말라.

사용자처럼 생각하기 위해 사용자와 함께 일하라.

구체적인 것보다 추상적인 것이 더 오래간다.

생각의 틀을 벗어나지 말고, 틀을 찾아라.

준비가 되었을 때 시작하라

어떤 일들은 설명하기보다 실제로 하는 것이 쉽다.

    - 명세의 순환에 빠지지 말라.

형식의 노예가 되지 말라.

팀을 기능 중심으로 조직하라.

테스트

    - 일찍, 자주, 자동
    - 모든 테스트가 통과하기 전엔 코딩이 다 된 게 아니다.

언어도 하나의 프로그래밍 언어인 것처럼 다루라.

문서를 애초부터 전체의 일부가 되게 하라.

사용자의 기대를 부드럽게 넘어서라

자신의 작품에 **서명**하라
